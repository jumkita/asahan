# 朝刊ニュース網羅確認システム — 要件定義（確定版）

最終更新: 2026-07-10

## 1. 背景・目的

| 項目 | 内容 |
|------|------|
| 課題 | 総合ニュース（経済寄りの比重）を毎日携帯で漏れなく確認するのが手間 |
| 目的 | 毎朝 6:30（JST）に、主要ソースの見出しをスマホで一通り確認できるようにする |
| 成功の定義 | PCを開けなくても、スマホのブラウザ（GitHub Pages）で「今日の朝刊相当」を一通り把握できる |

テンバガーDD とは無関係の独立プロジェクト。

## 2. 確定した制約・方針

| 項目 | 決定 |
|------|------|
| スコープ | 総合ニュース。経済カテゴリの比重をやや多め |
| 必須ソース | 日経。公式RSSが無い場合は Google News RSS（`site:nikkei.com`）＋日経ビジネス公式RSSで代替 |
| 通知 | **GitHub Pages（スマホ向けWeb）を主経路**。メールは任意 |
| 実行基盤 | **クラウド（GitHub Actions）**。PC電源不要 |
| 配信時刻 | 毎朝 **6:30 JST**（= 前日 21:30 UTC） |
| 曜日 | **土日も含む毎日** |
| リポジトリ | `C:\Users\jukit\Cursor\ニュース` |

## 3. スコープ

### In Scope
- 複数メディアの朝刊相当見出しの自動収集
- 重複排除（同一URL・類似タイトル）
- 経済寄りの並び（経済ソースを上部に厚め）
- スマホ向け Web サイト（GitHub Pages）への公開
- （任意）メールダイジェスト送信
- HTMLアーカイブの保存

### Out of Scope
- 有料記事全文の自動取得・スクレイピング
- AI要約の必須化
- 投資判断・テンバガーDD連携
- マルチユーザーSaaS

## 4. 機能要件

### Must
1. 指定ソースから見出し・要約（ある場合）・URLを取得する
2. 毎朝 6:30 JST にダイジェストを生成する（土日含む）
3. スマホ向けページ（GitHub Pages）で閲覧できるようにする
4. 日経関連見出しを必ず含める（取得失敗時は明示する）
5. 公式ページへのリンクを付ける（無料記事は「無料記事を開く」、会員壁があり得るものは明示）
6. 各記事に**要約（RSSにあれば）**またはローカル補足を付ける
7. RSS要約が薄い記事は、公式ページの**無料公開リード（description等）を取得して補足**する。有料本文は取得しない

### Should
8. URLベースの重複排除
9. ソース別セクション（日経 → 経済 → 総合の順）
10. 収集失敗ソースのエラー要約をページ末尾（および任意メール）に出す
11. （任意）SMTP 設定時はメールでも届ける

### Could（後続）
12. LINE / Discord への併用
13. 既読管理
14. キーワードハイライト

## 5. 情報源（Phase 1）

| ID | 名称 | 種別 | URL / 備考 | 比重 |
|----|------|------|------------|------|
| nikkei_gnews | 日経（Google News） | 必須 | `news.google.com/rss/search?q=site:nikkei.com&hl=ja&gl=JP&ceid=JP:ja` | 高 |
| nikkei_biz | 日経ビジネス | 公式RSS | `business.nikkei.com/rss/sns/nb.rdf` | 高 |
| nhk_main | NHK主要 | 公式RSS | `www.nhk.or.jp/rss/news/cat0.xml` | 中 |
| nhk_economy | NHK経済 | 公式RSS | `www.nhk.or.jp/rss/news/cat4.xml` | 高 |
| yahoo_top | Yahoo!ニュース主要 | 公式RSS | `news.yahoo.co.jp/rss/topics/top-picks.xml` | 中 |
| yahoo_business | Yahoo!ニュース経済 | 公式RSS | `news.yahoo.co.jp/rss/topics/business.xml` | 高 |
| mainichi_flash | 毎日新聞速報 | 公式RSS | `mainichi.jp/rss/etc/mainichi-flash.rss` | 中 |

※ 各メディアの利用規約を遵守。RSSは個人利用・見出し取得に限定。全文再配信は行わない。

## 6. 通知設計

- **主経路**: GitHub Pages（`site/index.html` を毎朝更新。スマホのホーム画面追加を推奨）
- **副経路（任意）**: SMTP メール（Gmail アプリパスワード等）。Secrets 未設定ならスキップ
- **件名例**: `[朝刊] 2026-07-10（金）見出し 42件`
- **本文**: HTMLメール。ソース別見出しリスト＋原文リンク
- **フォールバック**: メール送信失敗時は Actions を失敗扱いし、ログと生成HTMLを残す

## 7. 非機能要件

| 項目 | 目標 |
|------|------|
| 可用性 | 毎日生成成功率 ≥ 95% |
| 遅延 | 6:30 JST から 20 分以内に配信完了 |
| 情報量 | スマホで 5〜10 分で流し読みできる量（目安 30〜60 見出し） |
| 秘密情報 | GitHub Secrets / `.env` のみ |
| コスト | GitHub Actions 無料枠内を想定 |

## 8. システム構成

```text
GitHub Actions (毎日 21:30 UTC)
  → Python: RSS収集 → 重複排除 → HTML/テキスト生成
  → SMTPでメール送信
  → data/archive/ に当日分を保存
```

## 9. 受け入れ条件（Phase 1）

1. 手動実行（`workflow_dispatch`）でメールが届く
2. 日経関連セクションに1件以上ある、または失敗が明示される
3. 件名に日付と件数が入る
4. README に前提・Secrets・実行手順・期待結果がある
5. 最低1つの自動テストが通る

## 10. フェーズ

| Phase | 内容 |
|-------|------|
| 1（本実装） | RSS収集 + メール通知 + Actions 定時実行 |
| 2 | ソース追加（yaml編集）／失敗通知改善 |
| 3 | 重複統合の高度化・キーワード強調 |
